콘텐츠 추가/수정 → 앱 반영 워크플로우

신규 이야기 추가의 목표 표준은

develop-flow.md §3.2.1의

draft JSON → proposal row → 관리자 승인 → release sync → 앱 배포 흐름이다.

이 문서는 그 흐름에서 쓰는 JSON/이미지/seed 작업의 세부 규칙과, 기존 콘텐츠를

직접 수정할 때의 보조 절차를 정리한다.

문서 계층: 이 문서는 develop-flow.md를 보조하는 콘텐츠 운영 하위 문서다. real 공개 타이밍과 배포 판단은 develop-flow에서 결정하고, 여기서는 JSON/이미지/seed 작업 순서를 다룬다.

0. 현재 코드 기준 한 줄 요약

변경 종류 기준 소스 적용 타이밍
이야기 메타 assets/events/*.json → supabase/events/*.sql DB seed 적용 즉시
이야기 위치 각 event row의 landmark_code + assets/landmarks/landmarks.json seed-stories-characters 전에 필수
인물 메타 tools/seed/character_meta.json 생성물 make build-character-meta / make seed-characters
장면 원본 이미지 assets/story_images/<title>/scene_N.png 앱 번들 전
런타임 썸네일 assets/story_images_thumbs/<short_dir>/scene_N.jpg + index.json make thumbnails
장면 Storage fallback events.scene_image_paths 신규 이야기 표준 흐름에서는 proposal-scenes 업로드 후 승인 시 복사
앱 asset 목록 pubspec.yaml make update-pubspec-assets
퀴즈 각 event row의 quiz_questions → supabase/quizzes/quizzes_seed.sql apply-seeds-quizzes

중요한 현재 사실:

  • title에는 001 같은 번호 prefix를 붙이지 않는다. 화면에 보일 순수 제목만 쓴다.
  • 이야기 순서는 story_index가 결정한다.
  • 인물 필드는 characters를 쓴다. 예전 문서의 persons는 현재 JSON 포맷이 아니다.
  • canonical seed는 events.scene_image_paths를 채우지 않는다. 새 이야기의 배포 전

이미지 fallback은 proposal 승인 경로에서 채워지는 Storage path가 담당한다.

  • SceneAssetLoader는 장면 이미지를 로컬 번들에서 먼저 찾고, 없으면

scene_image_paths를 Supabase Storage public URL로 바꿔 fallback한다.

  • make release-sync-stories는 deleted_at IS NULL인 active events만 통합 JSON과 이미지

source로 삼는다. 삭제 승인된 row는 DB에 남아도 로컬 release 번들에서는 빠진다.

  • make thumbnails는 current assets/events/*.json에 등장하는 title만

story_images_thumbs/index.json에 넣고, story thumbnail 하위 디렉토리를 clean rebuild한다. 삭제된 title의 원본 폴더가 실수로 남아 있어도 앱 번들 썸네일로 다시 살아나지 않고, 중간 삽입으로 재사용된 <era>_<index> 폴더에 이전 이미지가 남지 않는다.

  • make release-sync-stories는 ensure-story-image-sources로 private

story-image-sources bucket에서 missing/changed 원본 PNG를 내려받고, 마지막에 upload-story-image-sources로 새 proposal 원본을 upsert하고 old manifest에만 남은 stale object를 삭제한 뒤 active manifest를 다시 올린다.

  • make apply-seeds-stories-characters는 Storage에 장면 이미지나 썸네일을 업로드하지

않는다. make upload-character-avatars도 캐릭터 아바타 전용이다.

  • 중간 삽입으로 기존 이야기들의 story_index가 밀리면 real DB에는 seed-only로

적용하지 않는다. 기존 events.id를 보존하는 DB shift+insert가 먼저 필요하다.

1. 신규 이야기 1건 추가

신규 이야기는 운영 DB에 바로 seed apply하지 않고, assets/story_drafts/*.json으로 초안을 만들고 proposal 승인 경로를 거치는 것을 표준으로 한다. 아래 직접 assets/events/*.json을 수정하는 절차는 release sync 결과를 검토하거나 기존 콘텐츠를 수정할 때의 기준 규칙으로 본다.

1.1 작업 전 동기화

로컬 JSON이 기준이다. 다만 real/dev DB에서 직접 수정한 내용이 있을 수 있으면 먼저 역추출한다. 이 명령은 로컬 파일을 덮어쓸 수 있으므로, 작업 중 변경이 있으면 먼저 diff를 확인한다.

git status --short assets/events
make export-stories-json ENV=dev

real DB를 기준으로 복원해야 할 때만:

make export-stories-json ENV=real

신규 이야기 표준 흐름에서는 이 역추출이 release sync 단계에 포함된다. 기존 콘텐츠를 직접 고치는 경우에는 저장소의 assets/events/*.json이 기준 소스다.

1.2 JSON 추가

release sync 후 canonical 파일을 검토하거나 기존 직접 편집 경로를 쓸 때는 해당 시대 파일에 새 객체를 추가한다. draft 단계에서는 같은 필드를 assets/story_drafts/YYYYMMDD_slug.json에 작성한다.

예:

{
  "title": "새 이야기 제목",
  "era": "era_divided_kingdom",
  "characters": ["elijah", "ahab"],
  "landmark_code": "lm_div_mt_carmel",
  "place_name": "갈멜 산",
  "lat": 32.731,
  "lng": 35.049,
  "summary": "한 문장 요약",
  "background_context": "시대 배경과 이 사건의 의미를 1~2문장으로 적는다.",
  "bible_ref": [{"book": "왕상", "from": "18:20", "to": "18:40"}],
  "start_year": -860,
  "end_year": -860,
  "time_precision": "approx",
  "story_index": 99,
  "unit_code": "div_elijah",
  "unit_title": "엘리야와 북이스라엘",
  "unit_order": 3,
  "story_scenes": [
    "장면 1 설명",
    "장면 2 설명",
    "장면 3 설명",
    "장면 4 설명"
  ],
  "scene_captions": [
    "사용자에게 보일 장면 1 설명",
    "사용자에게 보일 장면 2 설명",
    "사용자에게 보일 장면 3 설명",
    "사용자에게 보일 장면 4 설명"
  ],
  "scene_characters": [
    ["elijah"],
    ["ahab"],
    ["elijah"],
    []
  ],
  "quiz_questions": [
    {
      "type": "fact",
      "display_order": 0,
      "question": "엘리야는 갈멜 산에서 누구에게 결단을 촉구했습니까?",
      "choices": ["이스라엘 백성", "애굽 군대", "바벨론 포로"],
      "answer_index": 0,
      "explanation": "왕상 18:21 — '너희가 어느 때까지 두 사이에서 머뭇머뭇 하려느냐'"
    }
  ]
}

draft JSON에서는 story_index 대신 after_story_index를 쓴다. 예를 들어 after_story_index: 12는 같은 시대의 12번 이야기 뒤에 새 이야기를 넣는다는 뜻이다. 관리자 승인 시 DB가 뒤쪽 story_index를 밀고 실제 story_index를 정한다.

작성 규칙:

  • story_index는 같은 era 안에서 중복되면 안 된다.
  • 중간에 끼워 넣으면 뒤 이야기들의 story_index가 같이 조정되어야 한다.
  • 단, real에 이미 공개된 시대의 중간 삽입은 로컬 JSON 재번호 + seed apply만으로

처리하지 않는다. generated seed는 (era_id, story_index)를 UPSERT 키로 쓰므로, 기존 row id가 다른 이야기 내용으로 덮일 수 있다.

  • generated seed의 stale 정리는 deleted_at is null인 활성 row만 hard delete 한다.

삭제 제안 승인으로 soft-delete 된 row는 진행도/제안 이력 보존을 위해 남긴다.

  • story_scenes, scene_captions, scene_characters 길이는 맞춘다.
  • characters와 scene_characters에는 인물 code를 쓴다.
  • landmark_code는 assets/landmarks/landmarks.json에 존재하는 code를 쓴다.
  • quiz_questions는 1~3개를 넣고, 각 문항은 선택지 3개와 answer_index 0~2를 가진다.
  • 새 인물 code는 JSON에 먼저 넣어도 된다. 이후 build-character-meta가 meta에 반영한다.

1.3 위치 매핑 추가

make seed-stories-characters는 모든 이벤트에 landmark_id가 필요하다. 이제 매핑은 별도 위치 매핑 파일이 아니라 event row의 landmark_code가 담당한다. landmark_code는 assets/landmarks/landmarks.json에 존재해야 한다. 새 landmark가 필요하면 landmarks.json도 추가하고 make seed-landmarks를 실행한다.

1.4 인물 meta / seed 생성

make build-character-meta
make seed-stories-characters

확인할 것:

  • tools/seed/character_meta.json에 신규 인물이 들어갔는지
  • 신규 인물의 name, tagline, prompt, is_active_default가 의도와 맞는지
  • supabase/events/characters_seed.sql과

supabase/events/events_seed_part_*.sql diff가 의도한 범위인지

신규 인물의 prompt가 마음에 들지 않으면 tools/seed/build_character_meta_json.py의 override/hint를 보강한 뒤 다시 실행한다.

1.5 장면 이미지와 썸네일 생성

make generate-avatars      # 신규 인물이 있을 때
make generate-story-images # 새 이야기 장면 이미지 생성
make thumbnails
make update-pubspec-assets

동작:

  • 기존 아바타/장면 PNG가 있으면 skip한다.
  • 마음에 들지 않는 장면은 해당 assets/story_images/<title>/scene_N.png를 지우고

make generate-story-images를 다시 실행한다.

  • make thumbnails는 앱 런타임용 짧은 디렉토리

assets/story_images_thumbs/<era_slug>_<story_index>/와 index.json을 만든다.

  • story thumbnail 디렉토리는 누락 원본 검사를 통과한 뒤 매번 clean rebuild된다.

중간 삽입/삭제로 같은 <era_slug>_<story_index>가 다른 이야기를 가리키게 되어도 이전 JPG가 남지 않는다.

  • make update-pubspec-assets를 빼먹으면 새 썸네일이 앱 번들에 포함되지 않는다.

1.6 퀴즈를 같이 추가할 때

퀴즈는 같은 event row의 quiz_questions 배열에 넣는다. 그 다음:

make seed-quizzes

주의:

  • quiz_questions가 빠진 active event는 quizzes_report.json의 events_without_quiz에 나온다.
  • 해설 첫머리의 절 근거가 같은 event row의 bible_ref 범위 밖이면 빌드가 실패한다.

1.7 dev 적용과 검증

make apply-seeds-stories-characters ENV=dev
make apply-seeds-landmarks ENV=dev       # landmark가 바뀌었으면
make apply-seeds-quizzes ENV=dev         # 퀴즈가 있으면
make upload-character-avatars ENV=dev    # 신규 인물 아바타가 있으면
python3 tools/app/verify_asset_paths.py
python3 tools/seed/verify_polygons_contain_events.py
scripts/run_dev.sh

검증:

  • 지도에서 새 이야기가 올바른 region/landmark에 보이는지
  • 시대/방법, 인물, 이야기 탭에서 순서가 자연스러운지
  • 상세 화면에서 장면 이미지와 caption이 보이는지
  • 신규 인물 카드, 아바타, tagline이 맞는지
  • 퀴즈를 추가했다면 퀴즈 진입/정답/해설이 맞는지

2. real 배포

2.1 가장 중요한 주의점

신규 이야기를 real DB에 먼저 공개하면 즉시 사용자에게 보인다. 하지만 새 장면 이미지는 앱 번들에 들어가는 구조라서, 아직 새 앱을 설치하지 않은 사용자는 새 썸네일을 갖고 있지 않다.

현재 canonical seed는 events.scene_image_paths를 비워 둔다. 따라서 real DB를 먼저 공개하면 구버전 앱 사용자는 새 이야기를 보되 장면 이미지는 비어 보일 수 있다.

중간 삽입이면 이미지보다 먼저 event_id 보존을 확인한다. events seed는 (era_id, story_index) 기준으로 UPSERT하므로, 기존 5번 자리에 새 이야기를 넣고 뒤쪽 index를 모두 바꾼 SQL을 그대로 real에 적용하면 기존 5번 row의 id가 새 이야기 내용을 갖게 될 수 있다. 사용자 저장/읽기/퀴즈 기록은 event_id를 참조하므로, 운영 DB에서는 insert_event_at_position(...) RPC나 별도 patch로 기존 row를 먼저 shift+insert한 다음 seed와 로컬 JSON을 맞춘다.

2.2 직접 seed 공개 순서

신규 이야기는 proposal 승인 흐름을 우선한다. 아래 순서는 기존 콘텐츠 수정처럼 events.id가 새로 생기지 않거나, 새 앱 출시 이후에만 공개해도 되는 직접 seed 배포 경로다.

# 1. dev에서 검증 완료
scripts/run_dev.sh

# 2. real 앱 빌드 준비
scripts/run_real.sh
scripts/build_android_real.sh
scripts/build_ios_real.sh

# 3. Store 심사/출시 준비
# Play Console / App Store Connect 업로드

# 4. 출시 타이밍에 real DB 공개
make apply-seeds-stories-characters ENV=real
make apply-seeds-landmarks ENV=real       # landmark가 바뀌었으면
make apply-seeds-quizzes ENV=real         # 퀴즈가 있으면
make upload-character-avatars ENV=real    # 신규 인물 아바타가 있으면
scripts/run_real.sh

현실적으로 사용자가 모두 즉시 업데이트하지는 않는다. 신규 이야기에서 이미지 빈 상태를 피해야 한다면 직접 seed 공개가 아니라 develop-flow.md §3.2.1의 proposal 승인 흐름을 사용한다.

2.3 구버전 앱에도 이미지를 보여야 할 때

필요한 구조:

  1. 로컬에서 만든 장면 이미지를 public Supabase Storage에 업로드한다.
  2. real DB의 해당 이벤트에 events.scene_image_paths를 채운다.
  3. 구버전 앱은 Storage URL로 이미지를 본다.
  4. 신버전 앱은 로컬 번들 이미지를 먼저 보므로 Storage 호출이 줄어든다.

신규 이야기 표준 흐름에서는 make apply-draft/make apply-drafts가 draft 이미지를 proposal-scenes에 업로드하고, 관리자 승인 RPC가 그 경로를 events.scene_image_paths로 복사한다. 구버전 앱은 로컬 번들에 새 썸네일이 없어도 SceneAssetLoader의 Storage fallback으로 새 이야기 이미지를 불러온다.

make upload-character-avatars ENV=real은 characters 버킷만 다룬다. 이야기 장면 PNG/JPG 또는 assets/story_images_thumbs를 Storage에 올리고 events.scene_image_paths 를 채우는 target은 아직 없다.

직접 seed 배포 경로에는 아직 장면 이미지를 Storage에 올리고 events.scene_image_paths를 채우는 로컬 Make target이 없다. 따라서 직접 seed 배포를 계속 쓰는 경우의 선택지는 둘이다.

  • real 공개를 새 앱 출시 이후로 미룬다.
  • 별도 도구를 만든다: assets/story_images/<title>/scene_N.png를 Storage에 업로드하고

events.scene_image_paths = ARRAY['bucket/path/scene_1.png', ...]를 patch하는 운영 스크립트/Make target.

수동 SQL/Storage 업로드로도 가능하지만, 경로 오타나 cleanup 타이밍 실수가 나기 쉬워 반복 운영에는 권장하지 않는다.

2.4 실수로 real에 먼저 공개했을 때

바로 draft로 숨긴다.

update public.events
set status = 'draft'
where title in ('새 이야기 제목');

앱은 events_ordered view를 읽고, 이 view는 status='published'만 노출한다. fallback 준비 또는 앱 출시 타이밍이 되었을 때 다시 공개한다.

update public.events
set status = 'published'
where title in ('새 이야기 제목');

3. 기존 콘텐츠 수정

3.1 제목/요약/본문/좌표/장면 문구 수정

scripts/run_dev.sh

장면 문구를 바꿔 새 이미지가 필요하면 해당 PNG를 지우고 이미지를 재생성한다.

make generate-story-images
make thumbnails
make update-pubspec-assets

이미지가 바뀌면 앱 재빌드와 Store 배포가 필요하다.

DB에서 직접 토글할 수 있다.

update public.characters

where code = 'goliath';

make seed-characters는 일반 false 기본값으로 기존 런타임 설정을 끄지 않지만, 명시 비활성 예외 코드는 seed에서 다시 false가 될 수 있으므로 diff를 확인한다.
