데이터 흐름

DB row가 앱 화면에 표시되고, 사용자 액션이 다시 DB와 상태에 반영되는 경로입니다.

1

Supabase Row

events, eras, quizzes, notes, proposals, notifications 테이블에서 데이터를 읽습니다.

2

Repository

쿼리와 RPC 호출을 감싸고, Supabase 응답을 모델로 변환합니다.

3

Model

fromMap()으로 immutable Dart 데이터 객체를 만듭니다.

4

Controller

비즈니스 로직을 처리하고 state.copyWith()로 상태를 갱신합니다.

5

Provider Watch

화면과 위젯이 필요한 상태만 구독해 렌더링합니다.

6

사용자 지시

진행도, 노트, 저장 구절, 제안, 알림 읽음 상태가 다시 Repository로 흐릅니다.

Codex 작업 흐름

앞으로 Codex와 작업할 때의 표준 루프입니다. PDCA가 요청 시작부터 최종 보고까지 감싸고, TDD는 Do 단계 안에서 동작합니다.

Plan

계획과 영향 범위

요청을 도메인으로 나누고 $frontend, $backend, $data-pipeline 등 필요한 스킬과 검증 범위를 정합니다.

Do

테스트와 실행

기존 테스트를 확인하고, 새 기능/버그는 실패 테스트를 먼저 만든 뒤 기존 패턴으로 작게 구현합니다.

Check

변경과 검증 확인

git diff, format, analyze, test로 의도 밖 변경, 문서 누락, secret, 큰 파일 메트릭 변화를 확인합니다.

Act

반영과 보고

실패 원인을 다시 수정에 반영하거나, 통과 결과와 남은 리스크, 다음 단계를 보고합니다.

Docs

문서 동기화

바뀐 도메인에 따라 FRONTEND, BACKEND, TESTING, AGENTS 문서를 함께 갱신합니다.

6. 멈춤

커밋/푸시는 사용자 지시 후

Codex는 변경 완료 후 멈추고, 커밋해줘, 푸시해줘 지시가 있을 때만 진행합니다.
