앱

단일 Flutter 앱

main.dart가 .env와 ENV=dev|prod로 Supabase를 초기화하고, Firebase/FCM은 실패해도 앱이 계속 뜨도록 optional 초기화한다. app.dart는 StoryHomeScreen을 홈으로 붙이고 전역 글자 크기 스케일을 MediaQuery.textScaler에 주입한다.

Flutter 앱은 레이어가 분명하다. 모델은 행 변환, Repository는 Supabase 호출, Controller는 상태 전이, 화면/위젯은 한국어 UX와 지도 상호작용을 담당한다.

Widgets lib/widgets/** 지도(StoryMapPanel), 선택 패널(StorySelectionPanel), 사건 상세(EventDetailPage), 성경 리더, 프로필, 주간/매일 퀴즈, 알림, 제안 하위 위젯으로 나뉜다. 큰 파일은 part와 extension으로 분리한다.
Services lib/services/push_service.dart FCM 초기화, 토큰 등록/해제, 포그라운드 메시지 처리. Firebase 설정이 없으면 앱은 푸시 없이 진행한다.
핵심 사용자 흐름

학습 흐름

  • 홈에서 구약/신약과 시대를 고르고, 장소에서 시작 또는 인물과 걷기를 선택한다.
  • 장소 모드는 region/landmark를 선택하고, 인물 모드는 캐릭터를 고른 뒤 사건 타임라인을 본다.
  • 사건 상세는 요약, 장면 이미지, 성경 본문 읽기, 퀴즈, 감정 새김을 순서대로 연결한다.
  • 본문 읽기 + 퀴즈 + 감정 새김이 맞으면 진행도와 프로필 복습 데이터가 갱신된다.
운영/참여 흐름

제안, 알림, 푸시

  • 사역자 플래그 사용자는 새 이야기, 삭제, 일반 제안을 작성할 수 있다.
  • 관리자는 제안을 승인/거절하고, 승인 시 DB RPC가 story index shift와 quiz insert를 처리한다.
  • 알림은 개인 알림과 broadcast 알림이 분리되어 bell UI와 FCM 푸시로 이어진다.
  • 주간/매일 퀴즈는 별도 진행도 테이블과 push dispatch 함수가 있다.

Core content

시대, 인물, 사건

eras, characters, events, events_ordered, character_eras가 앱의 메인 목록과 정렬 기준이다. events.story_index는 era 내부 순서를 결정하고, events_ordered가 rank_in_era와 global_rank를 계산한다.

Map

위치 모델

landmarks는 region/anchor/minor/point를 통합하고, region polygon도 함께 가진다. 앱은 부팅 시 landmarks를 전체 로드하고 선택된 era code로 클라이언트 필터링한다.

User data

사용자 진행도

user_event_progress, user_quiz_attempts, user_event_emotion_marks, user_saved_events, user_saved_verses, weekly_quiz_progress, user_daily_quiz_attempts가 학습 기록을 나눠 가진다.

DB/RPC 영역 무엇을 담당하는가 앱에서 만나는 위치
성경 본문 bible_verses에 KRV 31,904절. SQL은 make seed-bible-verses로 생성한다. BibleReaderPage, 저장 구절 화면, 사건 상세의 본문 읽기.
제안 워크플로 event_proposals, event_proposal_comments, submit_event_proposal, approve_event_proposal, approve_delete_proposal, reject_event_proposal 등. 삭제는 soft delete이며 hard delete 금지. ProposalRepository, 제안 작성/상세/게시판 화면, proposal widgets.
알림/푸시 notifications, broadcast_notifications, user_push_tokens, list_my_notifications, unread_notification_count, send-push Edge Function. bell 버튼, 알림 드롭다운/히스토리, PushService.
Storage profile-images, characters, proposal-scenes, proposal-characters, proposal-general-images 버킷과 RLS 정책. 프로필 이미지, 캐릭터 아바타, 제안 이미지 생성/동기화.
Edge Functions generate-proposal-scene은 장면 이미지 생성, generate-proposal-character는 신규 캐릭터 아바타 생성, send-push는 FCM 발송. ProposalRepository.generateProposalScene, generateProposalCharacter, 알림 트리거.
백엔드 작업 시 가장 조심할 것

새 테이블은 RLS, grant, 필요한 index가 같이 필요하다. 이야기 삭제는 학습 진행도와 제안 이력이 얽혀 있으므로 deleted_at soft delete 흐름을 유지한다. 원격 DB 반영은 사용자가 명시적으로 요청하지 않는 한 하지 않는다.

개발자가 따를 작업 루프

실제 작업은 작게 자르고, 관련 테스트를 먼저 찾고, 바뀐 코드와 문서를 함께 움직인다.

1. Plan

도메인을 분류한다. UI면 $frontend, DB/RPC면 $backend, Python/Make/assets면 $data-pipeline, 테스트/CI면 $testing. 관련 문서와 테스트를 rg로 먼저 찾는다.

2. Do

새 기능/버그는 테스트를 먼저 맞춘다. 기존 패턴을 따르고, 모델은 불변, Repository는 주입된 SupabaseClient, Controller는 copyWith 상태 전이, UI 문구는 한국어, 디자인 토큰은 lib/theme/ 우선.

3. Check

git diff, 포맷, flutter analyze, focused flutter test, 필요 시 전체 flutter test, Python 도구는 python3 tools/run_unit_tests.py, asset/polygon/forbidden/code-metrics 검사를 본다.

4. Act

실패한 검증은 원인을 반영해 다시 수정한다. 실행하지 않은 검사는 통과했다고 말하지 않는다. commit/push/PR은 사용자가 명시적으로 요청할 때만 한다.
